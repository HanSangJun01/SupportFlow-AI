## 1. System Classification

**System Type:** Structured classification service

**Description:**
Phase 4 adds a separate FastAPI AI service that accepts tenant-scoped ticket content from the Spring backend and returns deterministic operational classification metadata: `category`, `urgency`, `sentiment`, `priority`, `confidence`, and `classifierVersion`. Good behavior means the service returns schema-valid, reproducible predictions for the same ticket content; the backend stores every classification attempt as a reviewable artifact; successful predictions automatically apply `category` and `priority` to the ticket; and every AI-applied workflow change remains traceable to the related `classificationAttemptId`.

**Critical Failure Modes:**
1. Cross-tenant leakage: a classification request, artifact, or ticket update exposes or mutates another tenant's data.
2. Silent mutation: a successful AI prediction changes ticket `category` or `priority` without a stored classification attempt and linked workflow history entry.
3. Failed attempt mutation: a timeout, unavailable AI service, validation error, or classifier failure changes ticket `category` or `priority`.
4. Contract drift: the AI service response omits required fields, returns invalid enum values, or returns confidence outside `0.0` to `1.0`.
5. Nondeterminism: the local classifier produces different outputs for the same ticket content, making tests and operational review unreliable.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** customer service / support operations

**User Population:** tenant operations managers, support leads, and support agents using backend APIs before authentication and frontend UI exist.

**Stakes Level:** Medium

**Output Consequence:** Successful classification immediately updates ticket `category` and `priority`, which affects triage, assignment focus, backlog review, and later operational metrics. It does not send customer-facing messages and does not close tickets.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|
| Category fit | Category label reflects the customer's actual issue type from subject/message keywords, such as billing, technical support, account access, cancellation, or general inquiry. | Category is unrelated to the inquiry, too broad when a clear keyword exists, or uses a label that downstream APIs cannot explain. | Misrouting and extra manual triage work. | `.planning/phases/04-ai-classification-integration/04-CONTEXT.md` |
| Priority fit | Priority escalates for urgent, blocked, outage, payment failure, security, or cancellation-risk language and stays lower for ordinary informational requests. | Priority is high for routine questions or low for urgent operational blockers. | Backlog ordering and support lead attention can be wrong. | `.planning/REQUIREMENTS.md` `FLOW-04`, `AI-02` |
| Sentiment fit | Sentiment reflects explicit frustration, anger, appreciation, or neutral wording without overreacting to single ambiguous words. | Sentiment misses strong frustration or marks neutral text as negative without evidence. | Poor prioritization signal for support leads. | Phase 4 discussion decisions |
| Traceability | Every successful auto-apply stores the attempt, result fields, classifier version, and a linked workflow history entry. | Ticket fields change without an artifact, actor/trigger, or `classificationAttemptId` link. | Audit gaps and loss of trust in automation. | `.planning/phases/02-tenant-workflow-core/02-CONTEXT.md` |
| Failure transparency | Failed attempts include status, trigger, timestamp, and error code/message; failed attempts leave ticket fields unchanged. | Failure is hidden, overwritten by defaults, or produces ticket field changes. | Operators cannot distinguish "not classified yet" from "classification failed." | `.planning/phases/04-ai-classification-integration/04-CONTEXT.md` |

### Known Failure Modes in This Domain

1. Keyword overmatch: a word like "cancel" in "cancel the previous request" could be mistaken for cancellation risk.
2. Priority inflation: too many tickets become high priority, making priority filtering useless.
3. Default category masking: fallback category looks successful but does not tell reviewers that only weak rules matched.
4. Artifact/history mismatch: a ticket shows updated category/priority but the attempt list or workflow history does not explain why.
5. Retry confusion: manual re-analysis creates a newer attempt, but consumers cannot identify which attempt last changed the ticket.

### Regulatory / Compliance Context

No formal regulated-domain obligation is introduced in Phase 4. The relevant constraints are product correctness constraints: tenant isolation, actor attribution where a human initiates re-analysis, append-only classification attempts, and no customer-facing final AI output.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Tenant operations manager | Reviews category/priority rubric and labels reference examples for realistic support workflows. |
| Support lead | Validates priority, urgency, and sentiment outcomes against triage expectations. |
| Backend/API owner | Verifies tenant isolation, response schema stability, artifact persistence, and workflow history traceability. |

## 2. Framework Decision

**Selected Framework:** FastAPI + Pydantic v2 direct service, with deterministic in-process rule classifier

**Version:** `fastapi==0.136.3`, `pydantic==2.13.4`, `pytest==9.0.3`; optional tracing package `arize-phoenix-otel==0.16.1` if monitoring hooks are implemented in Phase 4.

**Rationale:**
Phase 4 does not need an agent framework, RAG framework, model provider SDK, prompt chain, or LLM orchestration. The locked behavior is a local deterministic keyword classifier exposed through a separate Python HTTP service. FastAPI fits the service boundary because it gives request/response body validation, OpenAPI schema generation, response-model filtering, and direct in-process tests through `TestClient`. Pydantic v2 fits the contract because the classifier output is structured data with fixed enum-like values and constrained confidence. A direct service avoids unnecessary LangChain/LangGraph/CrewAI complexity and keeps local tests reproducible without provider credentials.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain | Overhead for a deterministic classifier; no prompt chain, tools, memory, or model calls are required. |
| LangGraph | Strong for stateful agent workflows, but Phase 4 has a single synchronous classification call and no graph state machine. |
| LlamaIndex | Optimized for retrieval and document Q&A; Phase 4 explicitly defers evidence retrieval to Phase 5. |
| CrewAI | Multi-agent role/task orchestration is unrelated to this deterministic classification service. |
| OpenAI Agents SDK | Provider-backed agent behavior is out of scope; Phase 4 requires no external LLM call or API key. |
| Pydantic AI | Useful for LLM-backed typed agents, but Phase 4 intentionally avoids LLM generation and keeps classification deterministic. |

**Vendor Lock-In Accepted:** No. Phase 4 uses open-source HTTP/API and validation libraries, not a model-provider-specific SDK.

### Key Abstractions

| Concept | What It Is | When You Use It |

| FastAPI path operation | Decorated async function that maps an HTTP route to Python code. | Expose `POST /api/v1/classifications/tickets`. |

### Common Pitfalls
1. Do not return unmodeled dict fields and assume clients will ignore them; FastAPI response models should filter output to the declared contract.
2. Do not use unconstrained floats for `confidence`; use `Field(ge=0.0, le=1.0)` and test boundary values.
3. Do not let defaults hide weak classification. Fallback predictions must have lower confidence and still include `classifierVersion`.
4. Do not include tenant data in logs or traces beyond non-sensitive IDs needed for correlation.
5. Do not introduce provider SDKs, prompt templates, embeddings, or async job queues in Phase 4.

## 4. Implementation Guidance

**Model Configuration:**
No LLM model is configured in Phase 4. The classifier is deterministic code. Use `CLASSIFIER_VERSION = "rules-v1"` or an equivalent explicit version constant and include it in every successful response and stored attempt. Keep rule data in code or a small local config file committed to the repo; do not load remote model/provider configuration.

**Core Pattern:**
Implement classification as a pure function:
1. Normalize `subject` and `customerMessage` by lowercasing and collapsing whitespace.
2. Evaluate ordered keyword groups for category, urgency, sentiment, and priority.
3. Return a `ClassificationResponse` Pydantic model with fixed enum values, a simple category string, rule-derived confidence, and `classifierVersion`.
4. Ensure fallback classification is explicit: category such as `general`, neutral sentiment, normal/low urgency, low/medium priority, and lower confidence.

**Tool Use:**
No AI tools or external provider tools are used. The only integration tool is HTTP between Spring backend and FastAPI service. Optional Phoenix tracing can wrap the classification function later, but it must not become required for correctness.

**State Management:**
The AI service is stateless. The Spring backend owns persistence for classification attempts, ticket category/priority mutation, and workflow history. The AI service must not persist tickets, attempts, tenants, or user data.

**Context Window Strategy:**
No model context window exists in Phase 4. Input bounds still matter: enforce request validation and reasonable maximum subject/message lengths in the API contract to prevent oversized payloads and noisy rule matching.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs - independent of framework choice.

### Structured Outputs with Pydantic

Use Pydantic as the contract source for both request and response. Keep `extra="forbid"` so unexpected client or classifier fields fail fast during tests.

```python
from enum import StrEnum
from pydantic import BaseModel, ConfigDict, Field

class Urgency(StrEnum):
    LOW = "LOW"
    NORMAL = "NORMAL"
    HIGH = "HIGH"
    CRITICAL = "CRITICAL"

class Sentiment(StrEnum):
    NEGATIVE = "NEGATIVE"
    NEUTRAL = "NEUTRAL"
    POSITIVE = "POSITIVE"

class Priority(StrEnum):
    LOW = "LOW"
    MEDIUM = "MEDIUM"
    HIGH = "HIGH"
    URGENT = "URGENT"

class ClassificationRequest(BaseModel):
    model_config = ConfigDict(extra="forbid", str_strip_whitespace=True)

    tenantId: str = Field(min_length=1)
    ticketId: str = Field(min_length=1)
    subject: str = Field(min_length=1, max_length=200)
    customerMessage: str = Field(min_length=1, max_length=5000)

class ClassificationResponse(BaseModel):
    model_config = ConfigDict(extra="forbid")

    category: str = Field(min_length=1, max_length=80)
    urgency: Urgency
    sentiment: Sentiment
    priority: Priority
    confidence: float = Field(ge=0.0, le=1.0)
    classifierVersion: str = Field(min_length=1)
```

Validation failures are contract failures in Phase 4. Do not add retry logic around deterministic classification; fix the rule output or schema instead.

### Async-First Design

Expose the FastAPI route as `async def` for standard ASGI compatibility, but keep the rule classifier itself synchronous and pure. Do not add background tasks or queues. The Spring backend calls the service synchronously with an approximately 2 second timeout and records success or failure attempts.

### Prompt Engineering Discipline

No prompts are used in Phase 4. If a later phase adds provider-backed classification, prompts must be introduced through a new AI-SPEC revision and eval plan, not silently inserted into this deterministic service.

### Context Window Management

No LLM context window is used. Treat input length validation as the equivalent safety control. Tests should include long but valid messages, empty/blank rejection, and fallback behavior for low-signal content.

### Cost and Latency Budget

Per-call provider cost is zero. Latency target is local HTTP plus deterministic rules. The FastAPI classification function should complete in milliseconds under normal local inputs; backend timeout default remains around 2 seconds to protect ticket creation and manual re-analysis.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|---------------------------|----------------------|----------|
| Contract validity | Pass if every successful response has `category`, `urgency`, `sentiment`, `priority`, `confidence`, and `classifierVersion`; enum values are fixed; confidence is within `0.0..1.0`; unexpected fields are rejected. | Code | Critical |
| Determinism | Pass if the same request body produces byte-equivalent classification fields across repeated calls and test runs. | Code | Critical |
| Category rule fit | Pass if reference examples with clear billing, technical, account access, cancellation, and general wording map to expected category strings. | Code + human-labeled fixture review | High |
| Priority/urgency escalation | Pass if urgent/blocking/security/payment-failure examples map to high or urgent priority and routine examples do not. | Code + human-labeled fixture review | High |
| Sentiment signal | Pass if explicit frustration maps negative, appreciation maps positive, and neutral operational requests remain neutral. | Code + human-labeled fixture review | Medium |
| Failure non-mutation | Pass if backend tests prove failed AI calls store failed attempts and do not change ticket category/priority. | Code | Critical |
| Traceability | Pass if successful backend classification stores an attempt and appends a workflow history entry linked to `classificationAttemptId`. | Code | Critical |
| Tenant isolation | Pass if cross-tenant reads/writes cannot access another tenant's classification artifacts or mutate another tenant's ticket. | Code | Critical |

### Reference Dataset

**Size:** Start with 15-20 examples.

**Composition:**
- 3 billing/payment examples, including failed payment and refund language.
- 3 technical support examples, including outage/blocking language.
- 3 account access examples, including password/login language.
- 2 cancellation/churn-risk examples.
- 2 positive/thank-you examples.
- 2 neutral/general inquiry examples.
- 2 low-signal or ambiguous fallback examples.
- 2 failure-path/backend integration scenarios for timeout/unavailable service.

**Labeling:**

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Request validation | Blank subject/message, oversized text, missing tenant/ticket ID, or unexpected fields. | Block with FastAPI validation error; backend records failed attempt only if the call reached the AI workflow. |
| Response validation | Classifier produces invalid enum, missing field, extra field, or confidence outside range. | Treat as AI failure; backend stores failed attempt and leaves ticket fields unchanged. |
| Backend timeout | AI service call exceeds approximately 2 seconds. | Ticket creation still succeeds; manual re-analysis returns structured failure; failed attempt is stored; category/priority unchanged. |
| Closed ticket mutation | Manual re-analysis requested for a closed ticket. | Reject before AI call; no new classification attempt unless planner explicitly models rejected requests separately. |
| Tenant mismatch | Ticket lookup does not match `{tenantId}` path or artifact access crosses tenants. | Return not found/denied according to existing backend tenant-isolation pattern; no AI call and no mutation. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|-------------------|-----------------------|
| Low confidence rate | Review all fallback-band attempts in development fixtures and sampled local runs. | Add or adjust keyword rules only with new fixture examples. |
| Manual re-analysis after auto-classification | Sample tickets where users request re-analysis after successful auto-classification. | Inspect category/priority misses and add labeled fixtures. |
| Failed attempt rate | Review every failed attempt in local/test environments. | Fix service availability, timeout config, or contract mismatch before execution verification passes. |
| Priority inflation | Compare count of HIGH/URGENT predictions against reference dataset expectations. | Tighten escalation keywords and add negative examples. |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix via OpenTelemetry (`arize-phoenix-otel`) is the default tracing option if Phase 4 includes tracing hooks. For the MVP, tests and stored artifacts are the source of truth; tracing must remain optional and must not require external credentials to run locally.

**Key Metrics to Track:**
- Classification request count by trigger (`AUTO_ON_CREATE`, manual re-analysis).
- Success/failure count by error code.
- Latency p50/p95 for AI service calls.
- Confidence band distribution.
- Auto-applied category/priority changes by classifier version.

**Alert Thresholds:**
- Any contract validation failure in tests blocks the phase.
- Any backend test where failed classification mutates category/priority blocks the phase.
- Any cross-tenant artifact access or mutation blocks the phase.
- Local p95 classification endpoint latency above 500 ms in deterministic tests should be investigated; backend timeout remains around 2 seconds.

**Smart Sampling Strategy:**
Prioritize human review for fallback-band attempts, urgent/high priority predictions, manual re-analysis results, failed attempts, and attempts made immediately before workflow metadata changes by a human.
