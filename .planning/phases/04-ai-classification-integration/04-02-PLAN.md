<objective>
Integrate the Spring backend with the AI classification workflow and append-only ticket artifacts.

User story: As a tenant operations manager and support lead, I want ticket creation and manual re-analysis to classify tickets, automatically apply successful category/priority predictions, and expose traceable classification artifacts through ticket APIs, so that SupportFlow AI reduces manual triage work without losing tenant-scoped auditability.

Purpose: Connect the Phase 4 FastAPI contract to the backend ticket workflow through a configurable HTTP client, embedded classification attempts, AI-applied workflow history, manual re-analysis endpoint, and targeted backend tests.
Output: Spring AI client/config, ticket classification attempt model, ticket service workflow changes, ticket response artifacts, manual re-analysis route, and backend unit/WebMvc coverage.
</objective>

<must_haves>
- D-06/D-07: Ticket creation runs classification synchronously and returns successful or failed classification artifact data.
- D-08/D-17/D-22/D-27: Failed classification attempts are stored but do not change ticket `category`, ticket `priority`, or workflow history.
- D-09/D-10/D-11: Manual re-analysis is allowed for any non-closed ticket and requires active same-tenant `actorUserId`.
- D-12: Automatic classification uses trigger metadata such as `AUTO_ON_CREATE` and no fake human actor.
- D-13/D-14/D-15: Classification attempts are append-only and embedded in ticket detail responses with attempt id, status, trigger, actor if manual, timestamps, success result fields, classifier version, and failure error code/message.
- D-16/D-18/D-19/D-20/D-21: Successful classification applies category/priority, appends a distinct `AI_CLASSIFICATION_APPLIED` history entry, and links that history entry to `classificationAttemptId`.
- D-28/D-29: Backend AI client uses a short configurable timeout around 2 seconds and no async retry jobs.
</must_haves>

<threat_model>
| Threat ID | STRIDE | Component | Disposition | Mitigation Plan |
|-----------|--------|-----------|-------------|-----------------|
| T-04-02-01 | Information Disclosure | Ticket classification attempts | mitigate | Store attempts only on tenant-scoped tickets retrieved with `findByTenantIdAndId`; expose attempts only through tenant-scoped ticket responses. |
| T-04-02-02 | Tampering | AI-applied category/priority | mitigate | Apply category/priority only after a successful validated AI response; failed attempts leave fields unchanged. |
| T-04-02-03 | Repudiation | AI workflow history | mitigate | Add `AI_CLASSIFICATION_APPLIED` history entries with `classificationAttemptId`; require `actorUserId` on manual re-analysis. |
| T-04-02-04 | Denial of Service | Backend AI HTTP call | mitigate | Use configurable timeout defaulting to approximately 2 seconds; no synchronous retries in Phase 4. |
| T-04-02-05 | Scope Creep | Async/RAG/draft behavior | mitigate | Do not add Redis jobs, retrieval, embeddings, draft generation, frontend, auth, or customer response behavior. |
</threat_model>
